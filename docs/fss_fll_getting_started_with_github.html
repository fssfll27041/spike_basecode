Section 01 — The Problem

The way we've been doing it has real costs.

Every practice, the same ritual: import the zip file from the thumb drive, code, then hope someone remembers to export a new zip before leaving. The code lives on a single USB stick — one misplaced drive away from chaos.

The failure modes are well-known: forgot to export at the end of practice; imported the wrong backup file; can't remember which zip is newest; the thumb drive is missing entirely.

This is a system problem, not a people problem. The system makes failure easy.
Section 02 — Source Control

A system that remembers everything.

Source control is software that tracks every change made to your files — who changed what, when, and why. It's not a backup; it's a complete, permanent history.

The cloud is the backup, automatically, always. There is no export step. There is no thumb drive. There is one source of truth, and it is always current.

If a mistake is made, you can roll back to any previous state. Experiment freely — nothing is ever permanently lost.

Each entry in the history (a commit) contains: a unique ID called a hash, a timestamp, the author's name, and a message explaining what changed and why. The diagram shows all four.

Section 03 — Git & GitHub

Git is the tool. GitHub is the place.

Git is software that runs on your laptop. It tracks every change you make to your code files. It works entirely offline — no wifi required. You get the full benefits of version tracking during every practice, even in the school library.

GitHub is a website that stores a shared copy of your team's code in the cloud. When a coach pushes your commits to GitHub after practice, your work is safe, backed up, and accessible to the whole team.

Git is the engine. GitHub is the garage where the team's car is parked.
Section 04 — Git Vocabulary

Learn the words. Understand the workflow.

Before diving in, here are the key terms you'll hear throughout this guide — each explained the way it actually feels to use it.

Repository
Your team's project folder — but one that remembers every version of every file, forever.
Commit
Like saving a file — but smarter. Records exactly what changed, who changed it, and why. You can always go back to any commit.
Push
Uploading your commits from your laptop up to GitHub. Your work joins the shared team copy in the cloud.
Pull
Downloading the latest commits from GitHub onto your laptop. Now you have everything your teammates pushed.
Branch
A separate line of work. The team uses one branch called main — that's all you need for now.
Merge Conflict
When two people edit the same part of the same file, git can't decide which version wins. The coach resolves it. Staying on your assigned mission file keeps this rare.
Tag
A permanent bookmark on a specific commit — used to mark the exact code that ran at a competition.
Section 04 — Student Workflow

At practice, students do two things.

Your team laptop arrives already configured. The repository is cloned, GitHub Desktop is set up, and your mission files are waiting. There is nothing to install or set up.

Your files live in one place on the laptop. When you open VS Code, you're editing those exact files directly — no copies, no confusion about which version is which. GitHub Desktop watches the same folder and notices your changes automatically.

01
EditOpen your assigned mission file and make your changes. Work freely — nothing you do here affects your teammates' laptops yet.
02
CommitWhen you reach a good stopping point, commit your work. Write a message that explains what you changed and why. This creates a permanent snapshot.

That's it for practice.Push to GitHub happens after practice, at home. That's the coach's job.
Section 05 — Coach Workflow

Coaches handle Push and Pull, at home.

The school library doesn't have wifi for team laptops. That's fine — Git works completely offline. Students commit throughout practice. After practice, at home, the coach connects and pushes everyone's commits to GitHub.

01
Push — after practice, at homeOpen GitHub Desktop on each laptop, connect to wifi, and push. All the commits made during practice go to GitHub.
02
Pull — before next practiceBefore the next session, pull on every laptop so each one has the full history of what every teammate did.
03
Merge if neededIf two students edited overlapping code, GitHub Desktop will flag it. The coach resolves it. With good file-per-mission discipline, this is rare.
Section 06 — The FLL Scenario

Practice without waiting in line.

During practice: Maya opens mission_two.py and refines her arm motor timing. Jonah opens mission_four.py and adjusts his gyro turn. Both commit when they hit good stopping points. No wifi needed. No waiting.

That evening: Coach connects both laptops to wifi at home and pushes to GitHub. Both students' commits are now safely in the cloud.

Before next practice: Coach pulls on every laptop. Everyone starts with the complete, merged history. No zip file. No thumb drive. No lost work.

Section 07 — Undoing a Commit

Made a mistake? You can fix it before it leaves your laptop.

If you commit something by accident — wrong file, bad change, incomplete work — you can undo it in GitHub Desktop before the coach pushes. Everything to the left of the push boundary is yours to fix.

In GitHub Desktop: right-click the commit in the History tab and choose "Undo." Your files go back to where they were before that commit, and you can start fresh.

Once a commit is pushed to GitHub, the coach handles any reversals. This is another reason push being coach-managed is a safety feature, not a limitation.
Only works before Push. Once the coach pushes to GitHub, the coach handles any fixes. That's why Push being the coach's job is a safety net.
Section 08 — History

Every change has an author, a date, and a reason.

GitHub Desktop's History tab shows every commit ever made — who did it, when, and the message they wrote. This is why commit messages matter. A message like "fixed stuff" is useless two weeks later. A message like "reduced arm speed from 80 to 60 for mission 3 — was overshooting" tells the whole story.

Good commit messages are a gift to your future self — and to every teammate who needs to understand what changed and why.

Section 09 — Competition Day

Sometimes one laptop is the right choice.

What does "parallelize" mean? It means multiple people working at the same time on different things — like two students coding on two laptops simultaneously. That's exactly what this workflow normally enables.

But on competition day, the team uses a single laptop. Not because git is hard — but because competition day is stressful, changes happen fast between runs, and this is not the moment to introduce git workflow steps that could cause confusion or mistakes.

One laptop. One source of truth. Everyone sees every change instantly. No merging needed afterward.

This is a deliberate process decision, not a step backward. Knowing when to simplify is just as important as knowing when to parallelize.

After the competition, the normal multi-laptop workflow resumes.

Section 10 — Competition Release

Tag the code after every competition.

After a competition — before resuming development — the coach creates a tag. A tag is a named, permanent bookmark on the commit history. It says: this is exactly what we ran at that event.

Naming convention:

format [team-number]-[team-name]-[event]-[year] examples 27041-thought-process-qualifier-2025
27041-thought-process-robonautica-2025

If scores drop after a later practice, the team can sync back to the competition tag, compare what changed, and isolate the regression. This is a real engineering workflow — and it works.

Section 11 — Ground Rules

Five rules that keep everything running smoothly.

01
One group per mission, per practice.Assign each mission to one group for the session — students often work in pairs. If two people are working on mission_three.py, they should share one laptop. Opening the same mission file on two laptops at once causes merge conflicts when the coach integrates.
02
Write a useful commit message.Your future self will thank you. "fixed stuff" tells nobody anything.
03
Pull before you start each session.Always begin with the latest code. Coaches pull before practice starts.
04
Push is the coach's job.Students commit; coaches push. This protects GitHub from accidental overwrites.
05
Tag after every competition.Before resuming development, the coach creates a tag. The competition run is preserved forever.
